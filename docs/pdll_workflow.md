# PDLL Workflow Summary

This document summarizes how PDLL (Pattern Descriptor Language) works in MLIR,
covering the build pipeline, C++ integration, native constraint APIs, and CLI
debugging tools.

---

## 1. PDLL → PDL → C++ Pipeline

### Build-time flow

```
MulToAdd.pdll                    # 1. High-level pattern definition (you write this)
    │  mlir-pdll -x=cpp          # 2. Compile to C++ at build time
    ▼
MulToAddPdll.h.inc               # 3. Generated C++ header containing:
                                      • Inline constraint C++ functions (auto-registered)
                                      • One struct per Pattern (embeds PDL IR text)
                                      • populateGeneratedPDLLPatterns()
    │
MulToAddPdll.cpp                 # 4. Pass implementation (you write this)
    │  calls populateGeneratedPDLLPatterns()
    │  calls registerNativeConstraints()
    ▼
applyPatternsAndFoldGreedily()   # 5. PDL pattern matching engine runs at runtime
```

### CMakeLists.txt key snippet

```cmake
add_mlir_pdll_library(MulToAddPdllIncGen
  MulToAdd.pdll
  MulToAddPdll.h.inc
)

add_mlir_library(MulToAdd
    ArithPatterns.cpp
    MulToAdd.cpp
    MulToAddPdll.cpp
    MulToAddShift.cpp
    ADDITIONAL_HEADER_DIRS
    DEPENDS
    MLIRMulToAddPasses
    MulToAddPdllIncGen          # ← depends on the PDLL generation target
    LINK_LIBS PUBLIC
)
```

This generates a build rule equivalent to:

```bash
mlir-pdll -x=cpp -I<includes> MulToAdd.pdll -o MulToAddPdll.h.inc
```

### Manual mlir-pdll CLI invocation (debugging / inspection)

`mlir-pdll` has three output modes (`-x` flag):

```bash
# 1. Generate C++ (what the build uses)
mlir-pdll -x=cpp -I /path/to/mlir/include MulToAdd.pdll -o MulToAddPdll.h.inc

# 2. Generate PDL MLIR (inspect the intermediate representation)
mlir-pdll -x=mlir -I /path/to/mlir/include MulToAdd.pdll

# 3. Generate AST (view the parse tree)
mlir-pdll -x=ast -I /path/to/mlir/include MulToAdd.pdll
```

**Notes:**

- `-I` **must** point to a directory containing MLIR TableGen definition files
  (e.g., `mlir/include/`), otherwise `#include "mlir/Dialect/.../...Ops.td"` in
  the `.pdll` file will fail.
- The `.pdll` file path is a **required positional argument**.
- `-o` specifies the output file; if omitted, output goes to stdout.

#### Example: Generate PDL MLIR for this tutorial

```bash
# From the mlir-tutorial root directory:
mlir-pdll -x=mlir -I ../llvm-project/mlir/include lib/Transform/Arith/MulToAdd.pdll
```

From the PDL MLIR output, you can observe the difference between inline and
external constraints:

```mlir
// Inline constraint IsPowerOfTwo (has [{...}] body) → pure assertion, no return value
apply_native_constraint "IsPowerOfTwo"(%1 : !pdl.attribute)

// External constraint Halve/MinussOne (declaration only) → returns a value bound to SSA
%8 = apply_native_constraint "Halve"(%1 : !pdl.attribute) : !pdl.attribute
```

This is exactly why `Halve` and `MinusOne`'s C++ implementations must use the
low-level API (needing `PDLResultList &` to push back return values), while
`IsPowerOfTwo` can use the high-level API (assertion only).

## 2. What You Write vs What Is Auto-Generated

| Component | Who writes | File |
|---|---|---|
| PDLL pattern definition (`.pdll`) | **You** | `MulToAdd.pdll` |
| Pass definition (`.td` + tablegen) | **You** | `Passes.td` |
| Pass struct (`runOnOperation`) | **You** | `MulToAddPdll.cpp` |
| External constraint implementation (e.g. `halveImpl`) | **You** | `MulToAddPdll.cpp` |
| Registration function (`registerNativeConstraints`) | **You** | `MulToAddPdll.cpp` |
| CMake build rules | **You** | `CMakeLists.txt` |
| `.h.inc` (inline constraint functions + Pattern structs + `populateGeneratedPDLLPatterns`) | **Auto** (by `mlir-pdll`) | `build-ninja/.../MulToAddPdll.h.inc` |
| Pass `.h.inc` (`MulToAddPdllBase`) | **Auto** (by `mlir_tablegen`) | `build-ninja/.../Passes.h.inc` |

### Generated vs hand-written interface contract

```
populateGeneratedPDLLPatterns(patterns);   // ← provided by .h.inc, fixed name
registerNativeConstraints(patterns);        // ← you write, any name you choose
applyPatternsAndFoldGreedily(...);          // ← MLIR framework built-in
```

## 3. Complete Pass Code Example (MulToAddPdll.cpp)

```cpp
#include "lib/Transform/Arith/MulToAddPdll.h"
#include "mlir/Dialect/Arith/IR/Arith.h"
#include "mlir/Dialect/PDLInterp/IR/PDLInterp.h"
#include "mlir/IR/PatternMatch.h"
#include "mlir/Transforms/GreedyPatternRewriteDriver.h"
#include "mlir/Pass/Pass.h"

namespace mlir::tutorial {

// Pass definition (generated by Passes.td tablegen)
#define GEN_PASS_DEF_MULTOADDPDLL
#include "lib/Transform/Arith/Passes.h.inc"

// ── External constraint implementations (low-level API) ──

LogicalResult halveImpl(PatternRewriter &rewriter, PDLResultList &results,
                        ArrayRef<PDLValue> args) {
  Attribute attr = args[0].cast<Attribute>();
  IntegerAttr cAttr = cast<IntegerAttr>(attr);
  int64_t value = cAttr.getValue().getSExtValue();
  results.push_back(rewriter.getIntegerAttr(cAttr.getType(), value / 2));
  return success();
}

LogicalResult minusOneImpl(PatternRewriter &rewriter, PDLResultList &results,
                           ArrayRef<PDLValue> args) {
  Attribute attr = args[0].cast<Attribute>();
  IntegerAttr cAttr = cast<IntegerAttr>(attr);
  int64_t value = cAttr.getValue().getSExtValue();
  results.push_back(rewriter.getIntegerAttr(cAttr.getType(), value - 1));
  return success();
}

// ── Register external constraints into the PDL engine ──

void registerNativeConstraints(RewritePatternSet &patterns) {
  patterns.getPDLPatterns().registerConstraintFunction("Halve", halveImpl);
  patterns.getPDLPatterns().registerConstraintFunction("MinusOne", minusOneImpl);
}

// ── Pass definition ──

struct MulToAddPdll : impl::MulToAddPdllBase<MulToAddPdll> {
  using MulToAddPdllBase::MulToAddPdllBase;

  void runOnOperation() override {
    mlir::RewritePatternSet patterns(&getContext());

    // Step 1: Load generated PDLL patterns.
    // populateGeneratedPDLLPatterns() is auto-generated by mlir-pdll in the .h.inc file.
    // It instantiates all Pattern structs (PowerOfTwoExpandRhs, etc.).
    // Each struct's constructor parses the embedded PDL IR text and
    // calls registerConstraintFunction("IsPowerOfTwo", ...) for inline constraints.
    populateGeneratedPDLLPatterns(patterns);

    // Step 2: Register external native constraints.
    // Before the PDL engine runs, bind the names "Halve"/"MinusOne" to C++ implementations.
    registerNativeConstraints(patterns);

    // Step 3: Run the greedy rewrite engine.
    // The engine calls registered constraint functions when matching patterns.
    // If it encounters an unregistered constraint name, it will error at runtime.
    (void)applyPatternsAndFoldGreedily(getOperation(), std::move(patterns));
  }
};

} // namespace mlir::tutorial
```

## 4. Generated .h.inc Structure (MulToAddPdll.h.inc)

Each PDLL Pattern generates one struct:

```cpp
// Inline constraints generate C++ functions directly
static ::llvm::LogicalResult IsPowerOfTwoPDLFn(::mlir::PatternRewriter &rewriter,
                                                ::mlir::Attribute attr) {
  int64_t value = cast<::mlir::IntegerAttr>(attr).getValue().getSExtValue();
  return success((value & (value - 1)) == 0);
}

// One struct per pattern
struct PowerOfTwoExpandRhs : ::mlir::PDLPatternModule {
  template <typename... ConfigsT>
  PowerOfTwoExpandRhs(::mlir::MLIRContext *context, ConfigsT &&...configs)
    : ::mlir::PDLPatternModule(::mlir::parseSourceString<::mlir::ModuleOp>(
      R"mlir(pdl.pattern @PowerOfTwoExpandRhs : benefit(2) {
        // ... PDL IR with apply_native_constraint "Halve"(%1) etc. ...
      })mlir", context), std::forward<ConfigsT>(configs)...) {

    // Auto-register only inline constraints (those with [{...}] bodies)
    registerConstraintFunction("IsPowerOfTwo", IsPowerOfTwoPDLFn);
    // NOTE: Halve and MinusOne have no body, so they are NOT registered here
  }
};

// populateGeneratedPDLLPatterns() iterates all structs and instantiates them
void populateGeneratedPDLLPatterns(::mlir::RewritePatternSet &patterns) {
  patterns.add<PowerOfTwoExpandRhs>(patterns.getContext());
  patterns.add<PowerOfTwoExpandLhs>(patterns.getContext());
  patterns.add<PeelFromMulRhs>(patterns.getContext());
  patterns.add<PeelFromMulLhs>(patterns.getContext());
}
```

### Key concepts

- **Inline constraints** (have `[{...}]` body): Both the C++ function and the
  `registerConstraintFunction("IsPowerOfTwo", ...)` call are generated by
  `mlir-pdll`. The user needs **zero** additional C++ code for them.
- **External constraints** (declaration only, no body): The PDL IR contains
  `apply_native_constraint "Halve"(%1)` references, but the user **must**
  manually implement and register them in C++, or the PDL engine will error at
  runtime.
- **Registration timing**: Constraint registration must happen **before**
  `applyPatternsAndFoldGreedily()` is called.
- **Symbol resolution analogy**: Like undefined symbols in an assembler — the
  PDL IR contains name strings, and `registerConstraintFunction` acts as the
  "link step" before execution.

### populateGeneratedPDLLPatterns() — who generates it?

This function is auto-generated by `mlir-pdll` in the `.h.inc` file. It
iterates over all PDL Pattern structs compiled from PDLL and generates one
`patterns.add<StructName>(...)` call for each.

```cpp
// ConfigsT is a variadic template for forwarding extra PDLPatternModule configuration
static void populateGeneratedPDLLPatterns(::mlir::RewritePatternSet &patterns,
                                          ConfigsT &&...configs);
```

## 5. PDLL Syntax Reference

### File structure

```pdll
#include "mlir/Dialect/Arith/IR/ArithOps.td"   // import MLIR dialect TableGen definitions

// ── Constraint definitions ──

// Inline constraint (with C++ body, auto-generated)
Constraint IsPowerOfTwo(attr: Attr) [{
  int64_t value = cast<::mlir::IntegerAttr>(attr).getValue().getSExtValue();
  return success((value & (value - 1)) == 0);
}];

// External constraint (declaration only, must implement in C++)
Constraint Halve(atttr: Attr) -> Attr;   // return value with -> Type syntax
Constraint MinusOne(attr: Attr) -> Attr;

// ── Pattern definition ──

Pattern PowerOfTwoExpandRhs with benefit(2) {   // higher benefit = higher priority
  // Match: match arith.muli whose LHS is arith.constant {value = const}
  let root = op<arith.muli>(op<arith.constant> {value = const: Attr}, rhs: Value);

  // Constraint checks
  IsPowerOfTwo(const);              // assertion constraint (check if const is power of 2)
  let halved: Attr = Halve(const);  // return-value constraint (returns const/2)

  // Rewrite
  rewrite root with {
    let newConst = op<arith.constant> {value = halved};
    let newMul = op<arith.muli>(newConst, rhs);
    let newAdd = op<arith.addi>(newMul, newMul);
    replace root with newAdd;        // replace matched root with newAdd
  };
};
```

### Key syntax rules

| Syntax | Meaning |
|---|---|
| `op<dialect.opname>(...)` | Match or create an operation |
| `{attrName = binding: Attr}` | Match a named attribute and bind to a variable |
| `lhs: Value` | Bind an operand to a variable |
| `Constraint name(args) [{ C++ }]` | Inline constraint definition |
| `Constraint name(args) -> Type;` | External constraint declaration (with return value) |
| `Pattern Name with benefit(N) { ... }` | Define a pattern; higher N = higher priority |
| `rewrite root with { ... }` | Rewrite body |
| `replace root with newOp;` | Replace the matched root operation with a new one |
| `let var = expr;` | Declare and bind a variable |
| `let var: Type = expr;` | Declare a typed variable |

## 6. Native Constraint API: Two Styles

### Inline constraints (have `[{...}]` body in .pdll)

- `mlir-pdll` auto-generates the C++ function and auto-calls
  `registerConstraintFunction()`
- User needs **zero** extra C++ code

### External constraints (declaration only, e.g. `Constraint Halve(attr: Attr) -> Attr;`)

- Must be manually implemented and registered in C++

#### Low-level API (for constraints that return values, e.g. Halve/MinusOne)

```cpp
// The signature is fixed: (PatternRewriter &, PDLResultList &, ArrayRef<PDLValue>)
LogicalResult halveImpl(PatternRewriter &rewriter, PDLResultList &results,
                        ArrayRef<PDLValue> args) {
  Attribute attr = args[0].cast<Attribute>();
  IntegerAttr cAttr = cast<IntegerAttr>(attr);
  int64_t value = cAttr.getValue().getSExtValue();
  results.push_back(rewriter.getIntegerAttr(cAttr.getType(), value / 2));
  return success();
}
```

- Return values are pushed via `results.push_back()`
- `PDLValue` must be manually `.cast<>()`-ed to the expected type

**Why is the signature so strange?**

- `args` is `ArrayRef<PDLValue>` (not concrete types like `Attribute`) because
  the PDL engine operates at the opaque handle level — it doesn't know the
  concrete C++ type of each argument.
- `PDLResultList &results` exists in both constraint and rewrite function type
  aliases (they share a single `using PDLConstraintFunction = ...`) to keep the
  dispatch mechanism uniform.
- For assertion-only constraints (no return value), `PDLResultList &` is
  ignored; for return-value constraints, results are pushed through it.

#### High-level API (for assertion-only constraints, no return value)

```cpp
// Signature: (PatternRewriter &, concrete types...)
LogicalResult isPowerOfTwoImpl(PatternRewriter &rewriter, Attribute attr) {
  int64_t value = cast<IntegerAttr>(attr).getValue().getSExtValue();
  return success((value & (value - 1)) == 0);
}
```

- `PDLResultList &` is discarded by the template wrapper, so it **cannot
  return values**
- Parameter types are automatically unwrapped (via `ProcessPDLValue<T>`
  specializations in the template machinery)

**How the high-level API works:**

`registerConstraintFunction` has a template overload that uses
`detail::pdl_function_builder::buildConstraintFn` to generate wrapper code:

1. Type-check each `PDLValue` via `ProcessPDLValue<T>::verifyAsArg()`
2. Unwrap `PDLValue` to the target C++ type via `ProcessPDLValue<T>::processAsArg()`
3. Invoke the user's type-safe function
4. Discard `PDLResultList &` (unnamed parameter), making return values impossible

### Selection guide

| Your constraint | Which API to use | Reason |
|---|---|---|
| Pure assertion (`IsPowerOfTwo`) | High-level (inline `[{...}]` or hand-written) | No return value needed |
| Return-value constraint (`Halve`, `MinusOne`) | Low-level (manual `PDLConstraintFunction`) | Need `PDLResultList::push_back()` to return results |

**Note:** Even if your constraint is inline (has a `[{...}]` body), if it
returns a value, `mlir-pdll` will auto-generate low-level form code. You never
need to worry about this.

### Type mapping table (PDLL ↔ C++)

| PDLL Type | C++ Type |
|---|---|
| `Attr` | `Attribute` |
| (derived attr) | `IntegerAttr`, `StringAttr`, etc. |
| `Op` | `Operation *` |
| `Type` | `Type` |
| `TypeRange` | `TypeRange` |
| `Value` | `Value` |
| `ValueRange` | `ValueRange` |
| (string) | `StringRef` (auto-unwrapped from StringAttr) |

## 7. Registration API Details

### registerConstraintFunction overloads

```cpp
// Low-level: register a function with the canonical signature directly
void registerConstraintFunction(StringRef name,
                                PDLConstraintFunction constraintFn);

// High-level (template): auto-wraps a typed function
template <typename ConstraintFnT>
void registerConstraintFunction(StringRef name,
                                ConstraintFnT &&constraintFn);
```

### Where to get the PDLPatternModule

```cpp
// From a RewritePatternSet:
patterns.getPDLPatterns().registerConstraintFunction("Halve", halveImpl);
```

`getPDLPatterns()` returns a reference to the internal `PDLPatternModule` that
manages the constraint function registry.
