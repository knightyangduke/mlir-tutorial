# How llvm-lit Works in This Project

## Running Tests

From the project root, always point into the **build** tree's tests directory:

```bash
# Run all tests
llvm-lit build-ninja/tests/

# Run a single test
llvm-lit build-ninja/tests/ctlz_runner.mlir
```

Do **not** run from the source `tests/` directory — that finds the Bazel config (`tests/lit.cfg.py`) which requires `RUNFILES_DIR` and will fail.

---

## How Config Discovery Works

When you invoke `llvm-lit <path>`, lit walks **up the ancestor chain** of `<path>` looking for a `lit.cfg.py` or `lit.site.cfg.py`. It does **not** search sideways.

- `build-ninja/tests/ctlz_runner.mlir` → walks up to `build-ninja/tests/` → finds `lit.site.cfg.py` ✓
- `build-ninja/test/foo` → walks up to `build-ninja/test/`, then `build-ninja/`, then root → never visits the sibling `build-ninja/tests/` ✗
- `tests/ctlz_runner.mlir` → walks up to `tests/` → finds `tests/lit.cfg.py` (Bazel config) → fails with missing `RUNFILES_DIR`

**Rule:** the argument must start with `build-ninja/tests/` so the ancestor chain passes through the directory containing `lit.site.cfg.py`.

---

## How Test Files Are Found (Source vs Build Tree)

`build-ninja/tests/` contains **no `.mlir` files** — only `lit.site.cfg.py`. The actual test files live in the source tree at `tests/`. Lit uses two separate roots:

- `test_exec_root = build-ninja/tests/` — where tests *run* (temp files, working directory)
- `test_source_root = /home/yeyang/mlir-tutorial/tests/` — where lit *reads* `.mlir` files from

When you pass `build-ninja/tests/ctlz_runner.mlir`:
1. Lit finds the config at `build-ninja/tests/lit.site.cfg.py`
2. Strips the suite root prefix: `build-ninja/tests/ctlz_runner.mlir` → `ctlz_runner.mlir`
3. Prepends `test_source_root`: `/home/yeyang/mlir-tutorial/tests/ctlz_runner.mlir`
4. Reads and runs that file

The argument path is a **virtual path** — it doesn't need to exist in the build tree, but the corresponding file **must** exist in `test_source_root`. (Confirmed: `llvm-lit build-ninja/tests/foo` fails with "contained no tests" if `tests/foo` doesn't exist in the source tree.)

---

## The Config File Chain

There are three config-related files, each with a distinct role:

### 1. `tests/lit.cmake.site.cfg.py.in` (CMake template, source tree)
Hand-edited template. CMake substitutes `@VARIABLES@` and writes the generated file to the build tree.

### 2. `build-ninja/tests/lit.site.cfg.py` (generated, build tree)
Auto-generated by CMake from the `.in` template. **This is the file lit discovers.** Its only job is to fill in build-specific paths and then delegate:

```python
config.llvm_tools_dir = "/home/yeyang/llvm-project/build-release/bin"
config.project_binary_dir = "/home/yeyang/mlir-tutorial/build-ninja"
config.project_source_dir = "/home/yeyang/mlir-tutorial"
# ...
lit_config.load_config(config, "/home/yeyang/mlir-tutorial/tests/lit.cmake.cfg.py")
```

### 3. `tests/lit.cmake.cfg.py` (hand-written, source tree)
The real config. Runs after the site config has set all the path variables:

```python
config.test_source_root = os.path.dirname(__file__)  # source tests/
config.test_exec_root = os.path.join(config.project_binary_dir, "tests")
config.suffixes = [".mlir"]
llvm_config.add_tool_substitutions(["mlir-opt", "mlir-runner", "tutorial-opt"], [...])
```

**Both files are always used** — `lit.site.cfg.py` first (provides paths), then `lit.cmake.cfg.py` (does the real configuration work).

---

## The Bazel Config (`tests/lit.cfg.py`)

This project also supports Bazel. `tests/lit.cfg.py` is the Bazel equivalent of `lit.cmake.cfg.py`. Key difference: Bazel passes tool paths via the `RUNFILES_DIR` environment variable at runtime, so no generated site config is needed.

| File | Build system | Entry point mechanism |
|---|---|---|
| `tests/lit.cfg.py` | Bazel | Discovered directly; uses `RUNFILES_DIR` env var |
| `tests/lit.cmake.cfg.py` | CMake | Loaded by the generated `lit.site.cfg.py` |

---

## Creating This Setup from Scratch

To replicate this setup in a new project, you need:

1. **`tests/lit.cmake.cfg.py`** — the hand-written main config (tool substitutions, suffixes, roots)
2. **`tests/lit.cmake.site.cfg.py.in`** — CMake template that captures build-dir paths and delegates to `lit.cmake.cfg.py`
3. **CMake wiring** in `tests/CMakeLists.txt`:
   ```cmake
   configure_lit_site_cfg(
     ${CMAKE_CURRENT_SOURCE_DIR}/lit.cmake.site.cfg.py.in
     ${CMAKE_CURRENT_BINARY_DIR}/lit.site.cfg.py
   )
   ```

CMake generates `build/tests/lit.site.cfg.py` at configure time, which becomes the lit entry point for the CMake build.
